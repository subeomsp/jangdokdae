## 1. 기술 피드백

### 1-1. 컴포넌트 선택과 한계

각 도구는 잘하는 영역이 다르므로, **작업 성격에 맞게** 배치해야 한다.

| 작업 성격 | 적합한 도구 | 이유 |
|---|---|---|
| 단순·정형 태스크 | Airflow | 스케줄링·재시도·DAG 관리에 특화 |
| 추론이 필요한 분기 | LangGraph (오케스트레이터 경유) | LLM 기반 판단·동적 흐름 제어 |

- 모든 흐름을 사전에 **케이스로 분기하는 것은 현실적으로 어렵다.** 예측 못한 입력을 전제로 설계해야 한다.
- **핵심 질문:** 컴포넌트 각각의 한계점을 명확히 정의했는가?

### 1-2. RAG 엣지케이스 처리

- 벡터DB에 없는 **예측하지 못한 데이터**가 사용자 질의로 들어오는 경우를 어떻게 처리할 것인가?
- 실제 서비스 레벨의 최적화까지는 가지 않더라도, **엣지케이스 처리 주체**를 정해야 한다.
  - Airflow에서 처리할 것인가, LangGraph에서 처리할 것인가?
- 이에 대한 백엔드 설계 근거가 필요하다.

### 1-3. 파이프라인 실패와 보상 로직 (가장 강조된 부분)

파이프라인 각 단계가 실패했을 때 **원복(rollback)이 가능하도록 보상 로직을 설계**해야 한다. (Saga 패턴 관점)

- 실패 지점 예시: 인스턴스 배포 → 컨테이너 배포 과정 중 이슈 발생 → 파이프라인 취소
- 취소 시 **두 관점을 분리**해서 다뤄야 한다.

| 관점 | 처리 내용 |
|---|---|
| 사용자(프론트) | "생성 중 오류가 발생했습니다" 등 안내 문구 출력 |
| 백엔드 | 생성 실패한 리소스를 삭제 (비동기 작업) |

- **각 단계마다** 실패 시 어떤 보상 로직이 돌아가야 하는지 정의가 필요하다.

### 1-4. 벡터DB 적재 시 카테고리화

- 한국투자증권 API, DART API → 벡터DB로 적재할 때 **카테고리화**를 거친다.
- 카테고리화 자체가 **1차 필터링** 역할을 한다.

### 1-5. 원문 처리 전략

- 뉴스 등 원문에서 **불필요한 요소는 제외**하고 **핵심 본문만** 남긴다.
- 처리 흐름: LLM 원문 요약 → **10줄 이내**로 축약 → 임베딩 → 적재.

### 1-6. LLM 출력 품질 검증

- 테스트 케이스를 추렸을 때, 원문의 **핵심 뉘앙스·문장**이 보존되는가?
- 향후 **사용자 질의응답에서 잘 검색되는가**가 품질의 핵심 기준이다.

### 1-7. RAG·오케스트레이터 호출 단위

- **한 번의 호출에는 하나의 task만** 담는다. (단일 책임 원칙을 호출 단위에도 적용)

### 1-8. 인프라

- 컨테이너 방식 배포는 하되, **클러스터 레벨까지는 가지 않을 것.**
  - 클러스터 레벨 = 자원 배치를 얼마나 효율적으로 할 것인가의 영역.
- 모델 파라미터 사이즈에 따라 **최적화 포인트**가 다르다 → **모니터링**으로 확인.
- 실사용자가 많아질 때 **분산**이 필요해진다. (현 단계에서는 과도한 최적화 불필요)

## 2. 사업 / 기획 피드백

### 2-1. 주린이 타깃 정의 (데스크 리서치 우선)

서비스 기획은 AI 설계가 아니라 **데스크 리서치부터** 시작해야 한다.

- **벤치마킹:** 유사 서비스 3~4개를 비교 분석한다. (설문이 가능하면 더 좋음)
- **산출물:** 페르소나, 유저 저니 맵, 화면 설계서, 화면 명세서, 기능 정의서 → AI로 작성 가능.
- **'주린이'는 너무 포괄적·주관적**이다. ("주식 경험 1~2년" 같은 기준도 모호함)
  - 모두를 위한 서비스는 이상해질 수밖에 없다 → 타깃을 단계적으로 좁혀야 한다.
- **독해력이 진입장벽**이 될 수 있다.
  - 종이신문을 접한 층과 아닌 층의 독해 수준이 다르다.
- **간이 설문 설계(5~10문항):**
  - "어느 정도를 주린이라고 생각하는가?"
  - 예: 20대 + 경제용어가 익숙치 않은 사람 → 용어를 아는지 **객관식**으로 측정.
- **시장 규모 정의:** TAM / SAM / SOM 으로 타깃 확장 경로 설정. ([참고](https://goodsharp.tistory.com/617))

### 2-2. 차별화 포인트와 Pain Point

공모전·포트폴리오에서는 **배경(내러티브)이 가장 중요**하다. 기술·내러티브는 AI 도움으로 그럴듯하게 만들 수 있지만, **Pain Point를 찾는 것이 핵심.**

- **벤치마킹 사례:**
  - 로빈훗(Robinhood): 주식창이 초단위로 널뛰어 "뭔지 몰라도 일단 들어가고 싶은" 시장에 진입.
  - 토스: **원래 있던 시장**에 뛰어들어 "포인트"로 사용자(파이)를 가져옴. (토스도 로빈훗을 참조)
- **분석 프레임워크 활용:**
  - 비즈니스 모델 캔버스
  - 린 캔버스
  - BCG 매트릭스 (이 분야의 캐시카우 서비스는 무엇인가?)
  - 5 Forces

### 2-3. 비즈니스 모델 — "AI가 사람을 얼마나 대체하는가"

- 모든 것은 결국 비즈니스 = **돈**이다. "어떻게 하면 사용자가 더 돈을 쓸까?"
- AI 상품 기획의 대중적 가치 = **사람이 하던 단순반복 업무를 비용효율적으로 대체.**
  - 예: 신한은행 인공인간 키오스크(어르신 대상 대면 영업점 대체), 보험사 해피콜(불완전판매 검증).
  - AI는 인건비를 올릴 필요도, 파업도, 노화도 없다.
- 장독대의 경우: **큐레이션·학습 서비스가 대체할 수 있는 직군**(예: 증권사 애널리스트)의 인건비 절감 효과를 수치화.
  - 단, "원래 많이 찾던 것"에서 가져와야 한다. (애널리스트 리포트는 원래 찾는 사람이 적었다 → 호응 부족)

### 2-4. 포트폴리오 / 면접 전략

- **혼자 여러 역할을 해봤다**는 점을 강조. "AI 도구를 활용해 여기까지 붙여봤고, 입사하면 이만큼 할 수 있다."
- **회사 맞춤 전략:** 지원하려는 회사를 타깃으로 설정 → 셀링 포인트를 담으면 포트폴리오를 이름만 갈아끼워 재사용 가능.
- **실패해도 인사이트를 정리**하라. "이 부분이 부족했고, 무엇을 보완하면 완결될 것 같다"를 GitHub에도 남긴다.
- **설명 능력:** 서류엔 다 담되, 면접에선 **문과 중학생도 이해할 정도로 쉽게·빠르게** 설명한다. 상급자가 의사결정할 수 있도록.
- **기획서가 완벽하면 개발은 문제없다.** 기획·아키텍처의 완성도가 우선.

## 3. 기술 스택 관련 (참고)

- 회사 내부에는 표준 개발 템플릿이 있고, 팀은 목적에 맞게 스택을 선정한다.
- 기술 스택은 잘 바뀌지 않고, **아키텍처를 추가·고도화**하는 방향으로 발전한다.
- 결론: 기획·아키텍처가 잘 되어 있으면 개발은 따라온다.
