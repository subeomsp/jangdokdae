## 1. 직무 · 커리어 (가장 강조된 부분)

### 1-1. 학습 능력 vs 운영 능력 — 무게추가 운영으로

- 모델 성능은 이미 최고점에 가까워졌다 → **직접 학습시키는 일은 줄고, 배포·서빙이 훨씬 많아질 것.**
- 결론적으로 **운영 능력이 우세**해지지만, **모델에 대한 이해도는 무조건 가져가야 한다.**
- GPU는 더 많이 쓰일 것이고, **GPU 엔지니어**가 주목받을 가능성이 크다.
- (박종환) 채용공고를 보면 **AI 서비스 배포·운영 직군**이 압도적으로 많다. 백엔드/인프라를 단순하게라도 — **이미지 빌드 → 컨테이너 배포 → 네트워크 → 쿠버네티스** 흐름으로 "어떻게 서비스를 띄울 것인가"를 다룰 줄 알아야 한다.

### 1-2. 신입에게 결국 중요한 건 "끝까지 낸 프로젝트 경험"

- 경력·학력 자체가 1순위 기준은 아니다. **팀이 필요로 하는 기술 스택을 가진 사람인지**가 먼저다.
  - 내 스택이 팀 요구와 정확히 맞는 것은 어느 정도 **운**이다 → 내 스펙을 원하는 팀을 찾아 지원하는 전략이 중요.
- **실제 서비스 운영 경험이 가장 중요.** 운영 경험의 최소 기준:
  - 최소 1명의 실사용자는 있어야 한다.
  - 대규모 트래픽 경험은 백엔드에 필요하지만, 서비스 엔지니어는 **동시 사용자 15~20명** 수준이면 충분히 의미 있다.
- 포트폴리오에는 **목표 → 이슈 → 해결 → 주요 성과**가 드러나야 한다.
- (MLOps 관점) **자동화 + 전체 라이프사이클을 한 바퀴 돌려본 경험**이 강한 변별력. 도구를 "써봤다"는 것만으로도 1차 필터를 통과한다.
- 가산점 요소: 쿠버네티스·도커·클라우드 경험, **AI 플랫폼(예: Lightning AI, Vertex)을 직접 써본 경험**. "우리 업무에 적합하겠다"는 인상만으로도 큰 가산점.

### 1-3. AI를 잘 쓰는 것도 실력 — 단, 코드를 이해해야 한다

- KT Cloud는 전 직원이 AI를 사용하고, **AI 직무 개발용 구독을 회사가 지원·권장**한다. 비개발자도 AI로 앱을 만든다.
- **AI가 뱉어낸 코드를 전부 이해할 수 있어야** 한다 → "AI를 잘 쓴다"의 진짜 기준.
- AI 플랫폼 직무는 **클라우드 역량 + 쿠버네티스** 위에서 동작 → GCP·Vertex 같은 사용 경험을 **포폴·프로젝트에 잘 녹여낼 것.**

### 1-4. 기본기는 시대가 바뀌어도 남는다

- 리눅스, 시스템 설계, 클라우드 자원에 대한 이해, 도커 컨테이너, 스케줄러 등은 **시대와 무관한 기본기.**
- 최신 기술이 있어도 그것을 받쳐줄 기본기가 없으면 의미가 없다.
- 트렌드 파악 채널(박종환): **링크드인 뉴스**(항상 옳진 않지만 트렌드용으로 좋음), 페이스북 PyTorch 페이지, 가짜연구소.

### 1-5. 면접관이라면 무엇을 보는가

- (박종환) **프로젝트 백그라운드**를 먼저 묻는다 → 기재된 자료가 허위인지, 현재 팀에 적합한지.
- (참고) 최지우 멘토는 KT Cloud AI 실무 면접관으로도 활동 중.

## 2. 모델 평가 · 품질

### 2-1. "모델 성능 평가"와 "서빙 성능 평가"는 다르다

- 인프라/플랫폼 관점(KT Cloud)에서는 외부 모델을 **중개·배포**하므로, 평가의 중심이 모델 정답률이 아니라 **배포 후 운영 안정성**이다.
- 핵심 서빙 지표:
  - **초당 처리 토큰 수(TPS, Tokens/Transactions per Second)**
  - **응답 지연(Latency)**
  - **배포 환경에서의 안정성** (직접 배포 시 서빙 엔진이 뱉는 메트릭을 직접 모니터링)
  - 그 외 운영 효율 지표: 전력 효율, TCO, 동시 사용자 수, 응답 정확도.
- (박종환) **환각(hallucination)에는 고정된 표준 지표가 없다.** 모델 성능 자체는 **논문 기반 지표**를 참고하고, 서비스 품질 보장은 **TPS** 위주로 본다.

### 2-2. LLM-as-a-Judge — 신뢰도의 전제는 "좋은 평가 모델"

- 신뢰성을 높이려면 **성능이 검증된 좋은 LLM을 평가 모델로 사용**하는 것이 우선.
- 단, 현업에서 **LLM이 LLM을 평가하는 것 자체는 크게 중요하게 보지 않는** 분위기.
  - LLM이 응답 속도 등을 한 번 평가·모니터링하고 알림을 주는 보조 용도로는 쓰지만, 절대적 지표로 신뢰하지는 않는다.
- 평가의 출발점은 항상 **"무엇이 좋은 것인가"를 정의하는 것.** 리트리버 정확도, context 관련성 등 기준이 광범위하므로 먼저 기준을 좁혀야 한다.

## 3. 모델 운영

### 3-1. 임베딩 모델 교체 — 결국 전체 재임베딩이 원칙

- 원본 텍스트를 저장해두거나 배치를 크게 묶어 **비용을 줄일 수는 있어도**, 결국 **전체 재임베딩은 피할 수 없다.**
- 교체 절차: **새 파이프라인을 배포**하는 것이 맞다.
- 무중단 교체는 기술적으로 가능하지만 **그런 도박은 하지 않는다.**
  - 새 버전이 준비되면 **10% → 30% → 50%** 식으로 품질 지표를 보며 점진 전환.
  - 이상이 감지되면 **롤백**할 수 있어야 하므로 단계적 전환이 안전하다.

### 3-2. 자체 GPU 서빙 vs Managed — 분기점은 "회사가 무엇을 요구하는가"

- 기업마다 편차가 크다.
  - GPU 인프라를 직접 관리하고 싶어 하는 기업 ↔ AI 모델만 관리하고 싶어 하는 기업.
- **모델 서비스 개발이 목적이라면 지금처럼 Vertex AI managed로 가는 것도 충분히 좋다.**
- 클라우드·인프라까지 직접 요구하는 회사라면 자체 서빙(GPU 최적화)을 검토.

### 3-3. 모델 선택 기준 (ML vs LLM, 상용 vs 오픈소스)

- 1순위: **당장의 서비스를 소화할 수 있는 모델**이 존재하는가.
- 적용 도메인이 한정적일 수 있으므로, **해당 task를 소화할 모델**이 있는지가 관건.
  - 모델을 직접 만들 경우 **데이터가 없는 것이 가장 큰 문제.**
- 둘 다 선택 가능하다면 **비용 측면에서 ML이 유리.**
- 상용화 결정은 아이디어 디벨롭보다 **임원 의사결정**이 크게 작용. (예: 동시 접속자 30명 같은 기준으로 판단)

### 3-4. 로컬/온디바이스 모델 수요

- "API 비용이 올라서 로컬 모델이 는다"는 단순 도식은 아니다.
- **온디바이스 수요**는 분명 존재하고, 기업 내부용 서비스에는 온디바이스 솔루션이 쓰이겠지만 **아직 많지 않다.**
- 로컬 모델 대중화에는 **인프라적 한계**가 있다.

## 4. 실무 함정 — 주니어의 과설계(over-engineering)

- 가장 흔한 과설계: **하이퍼파라미터 튜닝을 끝없이 반복**하거나, **논문을 구현해 성능을 잡는 데** 매달리는 것.
  - 성능을 잡긴 하지만, **결국은 욕심** — 더 효율적으로 가지 못하는 데서 비용이 발생한다.
- 기업이 원하는 AI 활용 수준: 거창한 게 아니라 **사내 업무에 지장 가지 않는 선**에서 실용적으로 쓰는 것.

## 5. 협업 · 라이선스 · 기술 선택

### 5-1. 팀 간 협업 (기획·서비스·인프라·보안·법무)

- 가장 중요한 태도는 **서로 존중할 수 있는 자세.**
- 기술적으로는 **필요한 데이터를 적재적소에** 전달하는 것, 그리고 **기획 문서·기능 명세**로 소통하는 것.

### 5-2. RAG 저작권 / 라이선스

- 기업보고서·재무제표를 임베딩해 벡터DB에 적재·검색에 활용하는 과정에서 **실제로 법적 문제가 발생한 적은 없었다.**
- (원문을 그대로 제공하지 않고 참조용으로만 쓰는 경우)

### 5-3. Robyn vs FastAPI — "최신이냐 검증이냐"의 선택 문제

- 운영에서 늘 반복되는 딜레마: **최신 기술 vs 검증되고 커뮤니티·자료가 풍부한 기술.**
- 대부분의 기업은 고객에게 **안정성**을 최우선으로 제공해야 하므로 검증된 기술을 선호 → **현재 단계에서는 FastAPI가 적절.**
- Robyn 같은 최신 프레임워크는 **기술적 차별성** 측면의 선택지지만,
  - 매 업데이트마다 운영 관점에서 적용이 어려울 수 있고, 자료가 적어 **어떤 문제가 터질지 예측이 어렵다.**
  - KT Cloud도 Robyn 도입 사례가 있으나, 최신 기술인 만큼 안정성 리스크를 안고 간다.
- 결론: **차별점은 최신 기술을 "조금씩" 들여다보며** 확보하되, 운영 기반은 안정적인 검증 기술로.

## 6. 장독대 프로젝트에 적용할 점

- **운영 경험을 만들 것.** 실사용자 1명 이상 + 동시 사용자 15~20명 수준의 데모/베타라도 돌려보면 "서비스 운영 경험" 서사가 생긴다. → [`docs/research/03-user-journey.md`](../research/03-user-journey.md) 기반으로 최소 사용자 확보 시나리오 정리.
- **라이프사이클을 끝까지 돌린 것을 강조.** 이미 `수집 → 전처리 → 임베딩·클러스터링`까지 Airflow 자동화를 완료했으므로, **AI 분석·번역 생성까지 한 바퀴**를 완주하는 것이 포폴 변별력의 핵심.
- **임베딩 모델 교체 전략을 문서화.** 전체 재임베딩 + 새 파이프라인 배포 + 10/30/50 점진 전환 + 롤백을 [`docs/evaluation/00-embedding-model-evaluation.md`](../evaluation/00-embedding-model-evaluation.md)에 마이그레이션 절차로 반영.
- **서빙/품질 지표를 분리해 정의.** "모델 성능"(논문 지표·정성 평가)과 "서빙 성능"(TPS·Latency·안정성)을 구분해 [`docs/evaluation/01-evaluation-checklist.md`](../evaluation/01-evaluation-checklist.md)에 명시.
- **자체 서빙은 보류.** 현재는 Vertex AI managed가 적절 — 모델 서비스 개발에 리소스를 집중하고 GPU 자체 서빙은 과도한 최적화로 두지 않는다.
- **과설계 경계.** 하이퍼파라미터/논문 구현에 매몰되지 말고, "사내 업무에 지장 없는 실용 수준"의 사고로 효율을 우선.
- **GCP·Vertex 사용 경험을 포폴에 명시적으로 노출.** AI 플랫폼 직무 가산점 요소이므로 README·아키텍처 문서에서 드러낼 것.
