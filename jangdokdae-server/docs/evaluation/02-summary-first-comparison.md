## 1. 한눈에 보기 (결론)

- **scope는 거의 안정(28건 중 1건만 변동)**이나, **frame은 표본을 키우자 6건(21%)이 바뀌었다** — 초판 n=5의 "frame 0/5
  완전 안정" 결론은 **표본이 작아 생긴 착시**였다. 요약이 가격 이면의 원인을 부각하며 PRICE→POLICY/PLAN/TREND로 재해석되는 경우가 다수.
- **기업 태그가 가장 크게 흔들린다 — 28건 중 11건(39%) 변동**, 그중 누락 우세(6건은 기업 수 감소). 요약이 그룹사·주변 기업을
  대거 쳐낸다(예: 331 `7→1`, 327 방산 `6→1`, 311 `16→6`). → **`company_ids` 백필 결과를 실제로 바꾸므로 통합 시 정합성 검증이 필수**다.
- **노이즈 제거 효과는 유효** — direction 3건이 더 신중해지고(부수 호재에 덜 휩쓸림), head 본문이 기사 본질부터 시작. 압축률 평균 **30%**(토큰 절감).
- 종합: **요약-우선은 토큰 절감·집중도 면에서 유망하나, "분류를 안 흔든다"는 초판 주장은 철회한다.** frame 21%·기업 39% 변동은
  무시할 수 없는 수준 → **분류 입력으로 쓰려면 기업 추출은 원문 병행** 등 안전장치가 필요(§5).

## 2. 실험 설계

- **Variant A (원문 기준선)** = 이미 적재된 `news_analysis`·`issue_docent`(현행 파이프라인 = 원문 본문 입력)를 그대로 사용.
- **Variant B (요약-우선)** = 대표기사 본문을 **요약 LLM**(Gemini, 핵심만 — 원인 사건·숫자·기업/기관명·호악재
  단서·근거 보존, 광고/상투구 제거)으로 요약 → `Issue.main_article.body`에 요약을 넣어 `classify` →
  `generate_with_guard` 재실행. `classify`·`generate` 모두 `body`만 보므로 입력이 요약으로 바뀐다.
  OPINION 현재가 보강은 A/B 공정 비교 위해 B도 `DataEnricher` 동일 적용.
- **대상**: 적재 클러스터 중 크기 큰 순 30건 → 본문 확보된 **28건**(2건은 본문 미확보/일시 연결오류로 스킵).
- 재현: `GOOGLE_CLOUD_PROJECT=<vertex> GOOGLE_APPLICATION_CREDENTIALS= uv run python -m scripts.compare_summary_first --limit 30`
  → 상세 `/tmp/summary_first_compare.json`.

**n=28 변화 집계**: scope 1 · frame **6** · direction 3 · sector 4 · companies **11**. 평균 압축률 30%(5~74%).

| cluster | 제목 | A(원문) → B(요약) |
|---|---|---|
| 334 | "외국인 돌아왔다"…삼전하닉 매수에 코스피 신고가 | PRICE → POLICY |
| 323 | '머니블랙홀'된 스페이스X…주가 70% 급등 | PRICE → PLAN |

> 예: cluster 29는 원문 head1이 부수 사실("미국·이란 종전 MOU…")로 시작한 반면, 요약본은 본질("Fed 금리 정책으로 이동…")부터 짚는다(direction 상승→중립).

## 4. 관찰

1. **scope는 안정, frame은 표본 확대로 흔들림이 드러남** — scope는 28건 중 1건만 변동했으나, frame은 6건(21%)이 바뀌었다.
   초판 n=5의 "frame 전건 동일"은 표본이 작았던 탓. 주로 원문이 가격 변동을 표제로 둔 `PRICE`에서, 요약이 원인 사건을
   부각하며 POLICY/PLAN/TREND로 재해석된다(§3.1). "어떻게 읽나"가 항상 보존되지는 않는다.
2. **방향이 더 신중해짐(노이즈 제거 효과, 유지)** — cluster 29: 원문 head1은 *"미국·이란 종전 MOU…"* 부수 사실로 시작했으나
   요약은 *"Fed 금리 정책으로 이동…"* 본질부터 짚고 direction `상승`→`중립`. 28도 동일 경향.
3. **기업 태그 변동이 가장 큼(39%) — 누락 우세** — 요약이 그룹사·주변 기업을 대거 정리한다(331 `7→1`, 327 `6→1`, 311 `16→6`).
   더 정확해지는 경우(301 중앙그룹 추가)도 있으나 누락이 우세(6 감소 vs 4 증가). **`company_ids` 백필이 직접 영향**받는다.
4. **압축률** — 짧은 기사는 효과 작고(최대 74%), 긴 기사에서 토큰·노이즈 절감이 크다(최소 5%). 평균 30%.

## 5. 결론 및 제안

**요약-우선은 토큰 절감(평균 30%)과 노이즈 제거(방향 신중화·본질 집중) 면에서 유망하나, 표본을 n=5→28로 키우자
"분류를 흔들지 않는다"는 초판 주장은 성립하지 않았다** — frame 21%·기업 태그 39%가 바뀐다. 채택하려면 이 변동을
관리할 안전장치가 전제다.

제안(우선순위순):
1. **(필수) 기업 추출은 원문 병행** — 요약이 그룹사·주변 기업을 누락해 `company_ids` 백필을 약화시킨다(§3.2).
   요약을 쓰더라도 **company_tags 추출만은 원문**으로 하거나, 요약 프롬프트에 "기업·계열사명 전수 보존"을 강제하고 누락 검출 시 원문 폴백.
2. **(검토) frame 재해석의 방향성** — 요약이 PRICE를 원인 frame으로 바꾸는 게 *개선*(가격 이면의 원인 포착)인지
   *드리프트*인지 사람이 표본 판정 필요. 개선이라면 오히려 요약을 분류 입력으로 쓸 근거가 된다.
3. **통합 시 설계** — 채택한다면 `prompts/news_summarize.yaml` + LangGraph `summarize` 노드(fetch→summarize→classify→generate).
   요약은 분류·생성 입력으로만, `evidence_spans` 원문 근거는 별도 처리.
4. **추가 표본** — n=28도 크기 큰 클러스터 편향이 있다. 다양한 크기·일자로 더 확대해 frame/기업 변동률을 안정화.

## 6. 사례별 상세 (판단 근거)


**A = 원문 본문**을 분류·생성 입력으로 쓴 현행 산출, **B = 요약 LLM이 추린 요약**을 입력으로 쓴 산출. 둘은 *같은 분류기/생성기*로 돌렸고 **입력만 다르다** → 차이는 곧 '요약이 결과에 미친 영향'이다.
원문 본문 전문은 저장돼 있지 않지만, **A의 4-head는 원문 기반 산출**이므로 *A vs (요약·B)*를 견주면 요약이 무엇을 보존·누락·왜곡했는지 판단할 수 있다.

판단 3축:
1. **분류 안정성** — scope·frame이 보존되는가(요약이 '읽는 법'을 바꾸면 위험).
2. **핵심 보존 vs 누락** — 요약이 기업·숫자·원인 등 핵심을 빠뜨리지 않는가(특히 **기업 누락**은 `company_ids` 백필을 약화).
3. **콘텐츠 품질** — B의 4-head가 본질부터 짚는가, A의 잘림/군더더기를 개선하는가, 거꾸로 **원문에 없던 내용으로 드리프트**하지는 않는가.

케이스마다 **판단 라벨**: **✅** 요약이 개선(노이즈↓·본질↑) · **⚠️** 누락/드리프트/분류이동(주의) · **⚠️혼재** 개선·주의 공존 · **=** 분류 동일(품질만 비교). *라벨은 안내일 뿐, 근거는 아래 4-head 전문이다.*
