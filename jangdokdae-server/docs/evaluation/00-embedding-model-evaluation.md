## 1. 한눈에 보기

장독대는 뉴스·사업보고서 텍스트를 숫자 벡터(임베딩)로 바꿔 **비슷한 것끼리 묶고(클러스터링)**, **질문에 맞는 문서를 찾는다(RAG 검색)**. 어떤 임베딩 모델을 쓸지가 이 두 기능의 품질을 좌우한다.

후보를 **최대 12종**까지 놓고 **세 가지 축**으로 평가한 결과, **`gemini-embedding-001`이 세 축 모두에서 1위**였다.

| 평가 축 | 무엇을 보나 | 1위 | 핵심 수치 |
|---------|------------|-----|----------|
| ① 비지도 클러스터링 (실험1) | 정답 없이 클러스터가 잘 뭉쳤나 | **gemini** | silhouette 0.515 |
| ② 라벨 클러스터링 (실험2) | 사람 정답대로 묶었나 | **gemini-001** | pair_auc 0.966 · ARI 0.307 |
| ③ RAG 검색 (실험3) | 질문에 맞는 문서를 찾나 | v2(0.966) > **gemini-001**(0.931) | recall@5 · MRR |

**확정값:**

```bash
EMBED_MODEL=gemini-embedding-001
EMBED_DIM=768
```

**부가 결론 1 — "금융 특화" 모델은 도움이 안 됐다.** 한국어 금융 도메인 특화 모델(nmixx-bge-m3·kf-deberta)을 일부러 후보에 넣었지만, 범용·관리형인 gemini를 이기지 못했다. 특히 nmixx-bge-m3는 자기 base 모델보다 처참하게 낮았다([§6](#6-실험3--rag-검색-품질--금융-특화-모델-검증) 참조).

**부가 결론 2 — 후속 모델 gemini-embedding-2(v2)는 축마다 갈림, 지금은 보류.** 올바른 호출 규약으로 재측정하니 **RAG는 v2가 001을 앞서고(0.966 vs 0.931)**, **클러스터링은 001이 명확히 우위(ARI 0.307 vs 0.155)**다. 다만 v2는 preview·us-central1 전용·배치 미지원·프롬프트 민감성 제약이 커 지금 도입은 보류한다([§7](#7-gemini-embedding-2-v2-비교--지금은-001-유지) 참조).

> **왜 중요한가:** 모델을 바꾸면 **저장된 임베딩을 전부 다시 계산**해야 하고, 차원이 다르면(예: 1024) **DB 스키마(`Vector(768)`)까지 마이그레이션**해야 한다. 그래서 **첫 배포 전에 한 번** 제대로 골라 못을 박는다.

## 2. 지표 사전 — 무엇을, 어떻게, 왜

아래 실험 결과표에 나오는 지표들의 정의다. **무엇을 재는지 / 어떻게 계산되는지 / 어떻게 읽는지**를 정리한다.

**먼저, 방향만 빠르게:**

| 지표 | 방향 | 범위 | "좋은 값"의 감 |
| --- | --- | --- | --- |
| pair_auc | ↑ 높을수록 좋음 | 0.5~1.0 | 0.95+ 우수, 0.5=무작위 |
| ARI | ↑ 높을수록 좋음 | 0~1 (음수 가능) | 우리 데이터선 0.2~0.3도 정상 |
| silhouette | ↑ 높을수록 좋음 | -1~1 | 금융 뉴스 0.2~0.35도 정상 |
| noise 비율 | ↓ 낮을수록 좋음 | 0~100% | 50% 초과면 과다 |
| recall@k | ↑ 높을수록 좋음 | 0~1 | 1.0=항상 top-k에 정답 |
| MRR | ↑ 높을수록 좋음 | 0~1 | 1.0=항상 1등으로 정답 |
| dim (차원) | 품질 아님 | 768 / 1024 | 성능 같으면 **작을수록(768) 운영 유리** |

각 지표의 상세는 아래에.

### 공통 — dim (차원)

- **무엇:** 임베딩 벡터를 이루는 **숫자의 개수**. `dim=768`이면 문장 하나가 768개 숫자로 표현된다.
- **왜 중요:** DB 컬럼이 `Vector(768)`로 고정돼 있어, **768 모델은 스키마 유지**·**1024 모델은 스키마 마이그레이션 + 전체 재계산**이 필요하다. 같은 성능이면 768이 유리.

### 클러스터링 지표 (실험1·2)

> 클러스터링 = 그날 들어온 뉴스 수백 건을 **같은 이슈끼리 자동으로 묶는 것**. 장독대는 `HDBSCAN` 알고리즘을 쓴다(밀집한 점은 클러스터로, 애매한 점은 -1=noise로 버린다). cosine = 두 벡터가 이루는 각도로 잰 유사도(1에 가까울수록 비슷).

#### pair_auc — 쌍 분리도 (★ 모델 간 비교 1순위)

- **무엇:** "같은 이슈 쌍"의 cosine이 "다른 이슈 쌍"보다 높게 나오는가.
- **계산:** 모든 쌍의 cosine을 점수로, 같은이슈=1·다른이슈=0 라벨에 대한 **ROC-AUC**.
- **읽기:** 0.5=무작위, 1.0=완벽 분리. **임계값을 정할 필요 없는** 단일 수치라 모델 간 공정 비교에 가장 적합.
- **한계:** pos/neg 쌍이 너무 쉬우면 모든 모델이 1.0에 수렴 → **하드 네거티브**(같은 템플릿·다른 이슈)가 변별의 핵심.

#### ARI (Adjusted Rand Index) — 정답 일치 (★ 파이프라인 직결)

- **무엇:** 모델 임베딩을 **실제 HDBSCAN에 통과시킨** 클러스터가 사람 정답(gold)과 얼마나 일치하는가.
- **계산:** `adjusted_rand_score(gold, hdbscan_labels)` — 우연 일치를 보정한 Rand Index.
- **읽기:** 0=무작위, 1=완전 일치. **값이 낮아 보여도 정상** — 150건 중 96건이 단독이라 0.2~0.3대가 정상 범위다. 절대값이 아니라 **모델 간 상대 순위**(gemini 0.307 > KURE 0.227)가 신호.

#### silhouette / noise 비율 (☆ 위생 체크)

- **silhouette:** 라벨 없이 클러스터가 내부적으로 뭉치고 서로 떨어졌는지(-1~1, ↑ 좋음, 금융 뉴스는 0.2~0.35도 정상). **각 모델이 자기 공간에서 계산**하므로 모델 순위 판정엔 약하고, "이 모델이 망가졌나"를 보는 용도.
- **noise 비율:** HDBSCAN이 어디에도 못 넣고 버린(-1) 기사 비율. silhouette과 **트레이드오프**(noise를 줄이면 응집이 약해짐)라 함께 본다.

### 검색(RAG) 지표 (실험3)

> RAG 검색 = 사업보고서를 잘게 쪼개(청크) 임베딩해 두고, **질문이 들어오면 의미가 맞는 청크를 찾아오는 것**.

#### recall@k — 상위 k 안에 정답이 들었나

- **무엇:** 질문을 던졌을 때, **정답 문서가 검색 결과 상위 k개 안에 들어오는 비율**.
- **예:** 질문 100개 중 정답이 top-5에 든 게 93개 → recall@5 = 0.93.
- **읽기:** `recall@1`은 "첫 번째 결과가 바로 정답인 비율"(가장 엄격), `recall@5`는 "5개 안에는 있는 비율"(느슨). k가 커질수록 값이 오른다. RAG는 보통 상위 몇 개를 LLM에 넣으므로 `recall@3`·`recall@5`가 실무 지표.

#### MRR (Mean Reciprocal Rank) — 정답이 얼마나 위에 오나

- **무엇:** 정답이 **몇 번째로 나오는지의 역수**를 평균. 순위 품질을 본다.
- **계산:** 정답이 1등이면 1점, 2등이면 1/2, 3등이면 1/3 … → 모든 질문의 평균.
- **예:** MRR 0.804면 정답이 평균적으로 **1~2등 사이**에 온다는 뜻. recall@5가 같아도 MRR이 높은 모델이 "정답을 더 위로" 올린다.

## 3. 평가를 세 단계로 나눈 이유

임베딩 모델은 **이미 훈련이 끝난 채로 다운로드**해 그대로 쓴다(frozen). 여기서 "비지도/지도"는 모델 학습 방식이 아니라 **결과를 채점하는 방식**이다 — 비지도 채점은 정답 없이 클러스터 모양만 보고, 지도 채점은 사람 정답(gold)과 대조한다.

| 단계 | 정답 라벨 | 역할 | 비용 |
|------|----------|------|------|
| 실험1 — 비지도 클러스터링 | 없음 | **방향성** — 후보를 빠르게 좁힌다 | 즉시 |
| 실험2 — 라벨 클러스터링 | 사람이 매긴 gold | **확정** — 정답 기준 최종 판정 | 라벨링 1회 |
| 실험3 — RAG 검색 | 사람이 매긴 쿼리-정답 | **다른 용도 검증** + 금융 특화 가설 검증 | 라벨링 1회 |

먼저 비지도로 넓게 훑어 후보를 좁히고, 라벨로 클러스터링을 확정하고, 마지막으로 RAG라는 다른 용도에서도 같은 결론이 나오는지 확인했다.

## 4. 실험1 — 비지도 클러스터링 비교 (10종)

### 설정

| 항목 | 값 |
|------|-----|
| 데이터 | 실제 수집 뉴스 제목 **387건** (`news`, `is_filtered=false`) |
| 방식 | 모델별 임베딩 → 동일 HDBSCAN → 지표 + t-SNE 시각화 |
| 도구 | [`scripts/embedding_experiment1.py`](../../scripts/embedding_experiment1.py) |

후보 10종(관리형 1 + 오픈소스 9): gemini, KURE-v1, KoE5, bge-m3-ko, bge-m3, Qwen3-Embedding, multilingual-e5-large, KR-SBERT, ko-sroberta(현 baseline), KoSimCSE.

### 결과 (silhouette 내림차순, 상위만)

> 방향: **silhouette ↑ 높을수록 좋음**, **noise 비율 ↓ 낮을수록 좋음**. (dim은 품질 아님 — 768이 운영 유리)

| 모델 | dim | 클러스터 수 | noise 비율 | silhouette |
|------|-----|------------|-----------|-----------|
| **gemini-embedding-001** | 768 | 89 | 31.5% | **0.515** |
| jhgan/ko-sroberta (baseline) | 768 | 85 | 38.5% | 0.484 |
| BAAI/bge-m3 | 1024 | 86 | 37.5% | 0.465 |
| nlpai-lab/KURE-v1 | 1024 | 90 | 34.4% | 0.460 |
| (이하 KoSimCSE·bge-m3-ko·KoE5·Qwen3·e5-large·KR-SBERT 0.41~0.45) | | | | |

- **gemini 단독 1위**(0.515) — 10종으로 넓혀도 gemini를 넘는 오픈소스 모델은 없었다.
- 정성 분석: gemini는 "삼성전자 시총 2000조"(9건)를 깔끔히 묶고, 증권사 상품(NH IMA·한투 엔화RP·유안타 펀드)을 각각 다른 클러스터로 정확히 분리했다. baseline은 `[더밸류 브리핑]` 5건을 **내용이 아니라 제목 템플릿**으로 잘못 묶는 약점을 보였다.

> **한계:** silhouette은 **각 모델이 자기 공간에서** 계산하므로 모델 간 절대 비교가 약하다(→ 실험2가 필요한 이유).

## 5. 실험2 — 라벨 클러스터링 (정식 판정)

### 왜 필요한가

"잘 뭉쳤다"(silhouette)와 "**정답대로** 묶었다"는 다르다. 그래서 **사람이 "이 두 기사는 같은 이슈"라고 매긴 정답지(gold)**를 만들어 모든 모델을 똑같이 채점했다. 이게 설계 §11이 요구하는 정식 판정이다.

### 라벨셋

| 항목 | 값 |
|------|-----|
| 뉴스 | 최신 **150건** ([`scripts/data/embedding_labelset.json`](../../scripts/data/embedding_labelset.json)) |
| gold 클러스터 | 18개 (소속 54건 / 단독 -1 96건) |
| `pos_pairs` (같은 이슈 쌍) | 49쌍 |
| `neg_pairs` (다른 이슈 쌍) | 50쌍 (이 중 **하드 네거티브 11쌍**) |

**하드 네거티브**(같은 템플릿·테마인데 다른 이슈)가 평가의 핵심이다 — 모델이 내용이 아니라 형식으로 묶는 약점을 잡는다. 예: `[더밸류 브리핑]` 3종(KB/자산운용/한투), 삼성 반도체 장비공급(예스티 vs 와이씨).

### 결과 (4종 — gemini v2 포함)

> 방향: **pair_auc · ARI · silhouette 모두 ↑ 높을수록 좋음**. 모델 순위는 pair_auc·ARI로 판정(silhouette은 위생 체크).

| 모델 | dim | pair_auc | ARI | silhouette | 판정 |
|------|-----|---------|-----|-----------|------|
| gemini-embedding-2-preview | 768 | 0.967 | 0.155 | 0.368 | (v2 비교 — [§7](#7-gemini-embedding-2-v2-비교--지금은-001-유지)) |
| nlpai-lab/KURE-v1 | 1024 | 0.971 | 0.227 | 0.364 | |
| **gemini-embedding-001** | 768 | 0.966 | **0.307** | **0.443** | ✅ **채택** |
| jhgan/ko-sroberta (baseline) | 768 | 0.915 | 0.213 | 0.337 | baseline |

- **pair_auc는 상위 3종이 박빙**(KURE 0.971 ≈ v2 0.967 ≈ 001 0.966). 셋 다 baseline(0.915)을 뚜렷이 앞선다.
- **ARI(정답 일치)는 001 단독 1위**(0.307). **v2는 클러스터링 instruction을 넣자 0.155로 baseline(0.213)보다도 낮아졌다** — v2는 프롬프트 형식에 매우 민감하고, 클러스터링용으로는 001이 명확히 낫다. ARI가 우리 파이프라인(같은 이슈 묶기)에 가장 직접적인 지표다.
- 001은 768차원이라 현 스키마를 유지하고, Vertex로 LLM 분석과 인프라가 통일된다. KURE를 쓰려면 1024 마이그레이션 비용이 든다. (v2 비교는 [§7](#7-gemini-embedding-2-v2-비교--지금은-001-유지).)

## 6. 실험3 — RAG 검색 품질 + 금융 특화 모델 검증

### 동기

실험1·2는 **뉴스 클러스터링**만 봤다. 다른 용도인 **사업보고서 검색(RAG)**에서도 같은 결론인지, 그리고 학계 주장("금융 도메인 특화 모델이 범용을 이긴다", FinMTEB EMNLP 2025)이 **한국어 사업보고서 검색**에서도 성립하는지 확인한다.

### 설정

| 항목 | 값 |
|------|-----|
| 코퍼스 | `report_chunks` **91청크** (6개 기업: 삼성전자·삼성물산·현대차·카카오·KB금융·하나금융지주, 섹션 단위) |
| 쿼리셋 | 사람 라벨 **29건** — 각 쿼리가 특정 기업의 특정 섹션을 정답으로 지목 ([`scripts/data/rag_queries.json`](../../scripts/data/rag_queries.json)) |
| 입력 캡 | 섹션 앞 **2000자** (서브청크) — 통짜(최대 89k자)는 일부 모델이 메모리 초과로 크래시 |
| 도구 | [`scripts/rag_recall_benchmark.py`](../../scripts/rag_recall_benchmark.py) |

코퍼스에 여러 기업의 동일 유형 섹션(원재료·재무건전성 등)이 섞여 있어, 모델이 **기업명 + 주제를 함께 변별**해 맞는 섹션을 위로 올리는지 본다.

### 결과 (recall@5 내림차순)

> 방향: **recall@1/3/5 · MRR 모두 ↑ 높을수록 좋음** (전부 0~1). recall@5는 "5개 안에 정답이 든 비율", MRR은 "정답이 평균 몇 등에 오나".

| 모델 | dim | recall@1 | recall@3 | recall@5 | MRR |
|------|-----|----------|----------|----------|-----|
| **gemini-embedding-2-preview (v2)** | 768 | **0.741** | **0.931** | **0.966** | **0.844** |
| gemini-embedding-001 | 768 | 0.672 | 0.897 | 0.931 | 0.804 |
| nlpai-lab/KURE-v1 | 1024 | 0.638 | 0.759 | 0.862 | 0.741 |
| BAAI/bge-m3 | 1024 | 0.603 | 0.793 | 0.862 | 0.728 |
| upskyy/kf-deberta-multitask (금융 특화) | 768 | 0.397 | 0.690 | 0.759 | 0.571 |
| jhgan/ko-sroberta (baseline) | 768 | 0.328 | 0.621 | 0.690 | 0.503 |
| **nmixx-fin/nmixx-bge-m3 (금융 특화)** | 1024 | 0.086 | 0.086 | **0.190** | 0.180 |

> **RAG에선 v2가 001을 앞선다**(recall@5 0.966 vs 0.931, 올바른 retrieval instruction 적용 시). 단 29쿼리 소표본이라 격차는 ~1~2건 수준. 클러스터링 결과(001 우위)와 운영 제약은 [§7](#7-gemini-embedding-2-v2-비교--지금은-001-유지).

### 핵심 — "금융 특화" 가설은 성립하지 않았다

1. **검색용으로 학습된 모델이 상위** — v2(0.966)·001(0.931)·KURE/bge-m3(0.862) 순. 관리형 gemini 계열이 한국어 검색 특화(KURE)·금융 특화 모델을 앞섰다.
2. **nmixx-bge-m3 붕괴(0.190) — "금융 특화"의 함정.** 자기 base인 `BAAI/bge-m3`(0.862)보다 처참하게 낮다. 사용 오류가 아니다: 모델 카드에 개발자가 직접 **"이거 금융 STS 말고 잘하는 거 없습니다"**라고 적어 두었다. 금융 **STS(문장 유사도)** 전용으로 좁게 튜닝돼 **검색 능력을 잃은** 모델이다.
   - **교훈:** "금융 특화" 라벨 ≠ 우리 과제(검색·클러스터링) 적합. **어느 과제로 특화됐는지**가 핵심이다.
3. **kf-deberta(0.759)는 baseline(0.690)만 이김.** 금융 LM 기반이라 baseline보단 낫지만 gemini·KURE·bge-m3에 못 미친다.

> FinMTEB의 "도메인 적응 우월" 주장은 영어/STS 중심 결과다. 한국어 사업보고서 **검색**에서는, 검색용으로 학습된 범용·관리형(gemini)·한국어 검색 특화(KURE)가 STS 특화 금융 모델을 앞섰다.

## 7. gemini-embedding-2 (v2) 비교 — 지금은 001 유지

### 왜 비교했나

`gemini-embedding-2`는 001의 후속(2026-03)으로 **컨텍스트 8192토큰(4배)**·멀티모달·MRL 768 절단을 제공한다. 프로덕션 임베딩을 아직 한 번도 안 한 **지금이 재임베딩 비용 0인 교체 적기**라, 라벨 클러스터링(실험2)·RAG(실험3) 동일 조건으로 001과 맞붙였다.

### 호출 규약 주의 — v2는 task_type을 무시한다 (검증된 함정)

v2는 001과 달리 **`task_type` 파라미터를 완전히 무시**한다 — 같은 텍스트를 `RETRIEVAL_QUERY`/`RETRIEVAL_DOCUMENT`로 임베딩해도 **벡터가 100% 동일**(cos=1.000)했다(001은 0.84로 정상 구분). 즉 task_type으로 부르면 쿼리·문서가 **대칭 임베딩**되어 검색 성능이 무너진다.

대신 v2는 **프롬프트에 task instruction을 직접** 붙여야 한다(Vertex 문서). 검색은 비대칭 형식을 쓴다:

- 쿼리: `task: search result | query: {text}`
- 문서: `title: ... | text: {text}`

> 이 차이를 처음엔 놓쳐 v2 RAG가 0.776으로 나왔으나(쿼리·문서 대칭 버그), **올바른 instruction으로 재측정하니 0.966으로 뒤집혔다.** [`embedding_client.py`](../../services/embedder/embedding_client.py)가 우리 `task_type` 추상화를 v2 instruction으로 번역하도록 수정했다.

### 결과 — 검색은 v2, 클러스터링은 001 (혼합)

| 지표 | 방향 | gemini-embedding-001 | gemini-embedding-2 | 승자 |
|------|------|------|------|------|
| 클러스터링 pair_auc | ↑ | 0.966 | 0.967 | 무승부 |
| 클러스터링 **ARI** (정답 일치) | ↑ | **0.307** | 0.155 | **001 (명확)** |
| 클러스터링 silhouette | ↑ | **0.443** | 0.368 | **001** |
| **RAG recall@5** | ↑ | 0.931 | **0.966** | **v2 (소폭)** |
| RAG MRR | ↑ | 0.804 | **0.844** | **v2** |

- **RAG 검색은 v2가 001을 앞선다**(recall@5 0.966 vs 0.931, MRR 0.844 vs 0.804). 단 29쿼리 소표본이라 ~1~2건 차이 — "명확한 우위"보다 "동급~소폭 우위".
- **클러스터링은 001이 명확히 낫다**(ARI 0.307 vs 0.155). v2는 클러스터링 instruction을 넣자 ARI가 **baseline(0.213)보다도 낮게** 떨어졌다 — v2는 프롬프트 형식에 매우 민감해, task별로 instruction을 튜닝해야 제 성능이 난다(001의 task_type은 turnkey).

### 운영 제약 — 지금 도입은 시기상조

1. **preview 상태** — `gemini-embedding-2-preview`. GA·SLA·가격 불확실, 프로덕션 리스크.
2. **리전 제약** — **us-central1 전용**. 우리 운영 리전 `asia-northeast3`엔 없다(404) → 한국 금융 데이터의 크로스리전 호출(지연·데이터 레지던시) 문제.
3. **배치 미지원** — embedContent가 **콘텐츠 1건씩만** 받는다. 분당 할당량도 낮아 대량 임베딩 시 429가 빈발(본 실험도 백오프 재시도로 우회).
4. **호출 규약 변경** — `task_type` → 프롬프트 instruction. 채택 시 [`embedding_client.py`](../../services/embedder/embedding_client.py) 재작업 필요(이미 v2 분기는 추가됨).

### 결론

**지금은 001 유지 — 단, "v2가 더 나쁘다"가 아니라 "001이 더 적합·준비됨"이라서다.**

- 001은 **일일 핵심 파이프라인인 뉴스 클러스터링에서 명확히 우위**(ARI 0.307 vs 0.155)이고, **turnkey**(task_type 견고, 프롬프트 튜닝 불필요)다.
- v2의 RAG 우위는 **소폭·소표본(29쿼리)**이고, **프롬프트 민감성**(instruction 형식에 따라 성능 급변)·**운영 블로커**(preview·us-central1 전용·배치 미지원)가 상쇄한다.
- **향후 보고서 RAG가 핵심이 되고** v2가 **GA + asia-northeast3**로 오면, **RAG 전용으로 v2를 재고**(또는 클러스터링 001 / RAG v2 하이브리드)할 가치가 있다. 코드엔 v2 분기가 이미 있어 모델명만 바꿔 재평가 가능하다.
