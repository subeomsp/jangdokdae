## 30초 요약

AI 에이전트 평가는 **정해진 과제를 에이전트에게 시키고, 결과가 성공 기준을
만족했는지 반복 측정하는 자동화된 테스트**다.

핵심만 다섯 줄로 줄이면 다음과 같다.

1. 평가는 에이전트가 복잡해진 뒤가 아니라 **개발 초기에** 만든다.
2. 처음부터 수백 개가 필요하지 않다. 실제로 자주 확인하던 **20~50개 과제**로 시작한다.
3. 평가는 코드, AI 모델, 사람 중 하나만 믿지 않고 **서로 다른 평가자를 조합**한다.
4. 모델 결과는 실행할 때마다 달라지므로 같은 과제를 **여러 번 실행**해 성공률과 일관성을 본다.
5. 점수만 보지 말고 실패 기록을 읽어 **에이전트가 틀린 것인지 평가 기준이 틀린 것인지** 확인한다.

## 1. 왜 필요한가

평가가 없으면 다음과 같은 개발이 반복된다.

```text
운영에서 문제 발견
→ 그 사례만 수정
→ 다른 사례가 망가짐
→ 다시 운영에서 발견
```

평가가 있으면 변경 전후에 같은 과제를 실행하여 개선과 회귀를 구분할 수 있다. 또한
“잘 작동해야 한다”라는 모호한 요구사항을 구체적인 성공 조건으로 바꾸게 한다.

## 2. 평가의 기본 구조

```text
평가 스위트
  └─ Task: 입력과 성공 기준이 정해진 과제
      └─ Trial: 같은 Task를 한 번 실행한 것
          ├─ Transcript: 출력, 도구 호출, 중간 결과 전체 기록
          └─ Outcome: 실행이 끝난 뒤 실제로 만들어진 최종 상태
      └─ Grader: 결과에 점수를 매기는 평가자
```

| 용어 | 뜻 |
| --- | --- |
| Task | 입력과 성공 기준이 있는 테스트 한 건 |
| Trial | 같은 Task를 실제로 한 번 실행한 결과 |
| Transcript | 프롬프트, 응답, 도구 호출, 중간 상태를 포함한 전체 기록 |
| Outcome | 에이전트가 말한 결과가 아니라 DB·파일 등에 실제로 남은 최종 상태 |
| Grader | 결과의 특정 측면을 채점하는 코드·모델·사람 |
| Evaluation suite | 비슷한 역량을 측정하는 Task들의 묶음 |
| Evaluation harness | Task 실행, 기록, 채점, 집계를 담당하는 실행 환경 |

예를 들어 에이전트가 “환불이 완료됐습니다”라고 답했다고 성공한 것은 아니다. 실제 주문
DB에 환불 상태가 반영됐는지를 확인한 것이 `Outcome` 평가다.

## 3. 능력 평가와 회귀 평가

### 능력 평가

“이 에이전트가 이 일을 얼마나 잘할 수 있는가?”를 측정한다.

- 처음에는 점수가 낮아도 된다.
- 어려운 사례를 포함한다.
- 프롬프트·모델·도구를 개선할 방향을 알려준다.

### 회귀 평가

“기존에 잘하던 일을 여전히 잘하는가?”를 측정한다.

- 이미 해결한 사례를 모은다.
- 거의 100%에 가까운 통과율을 기대한다.
- 코드, 프롬프트, 모델 변경으로 기존 기능이 망가지는 것을 막는다.

처음에는 능력 평가였던 과제가 안정적으로 통과되기 시작하면 회귀 평가로 옮길 수 있다.

## 4. 세 가지 평가자

### 코드 기반 평가자

정확한 문자열, JSON 구조, DB 상태, 도구 호출, 실행 시간 등을 코드로 확인한다.

- 장점: 빠르고 저렴하며 결과가 항상 동일하다.
- 단점: 자연스러움이나 미묘한 의미 차이를 판단하기 어렵다.
- 원칙: 코드로 확인할 수 있는 것은 최대한 코드로 검사한다.

### 모델 기반 평가자

별도 LLM이 루브릭을 보고 정확성, 근거 충실도, 자연스러움 등을 판단한다.

- 장점: 자유로운 문장과 의미 차이를 평가할 수 있다.
- 단점: 비용이 들고 평가 결과도 매번 달라질 수 있다.
- 원칙: 기준을 여러 항목으로 나누고, 정보가 부족하면 `Unknown`을 선택할 수 있게 한다.

### 사람 평가자

도메인 전문가나 제품 담당자가 직접 검토한다.

- 장점: 최종 사용자와 가장 가까운 품질 판단이다.
- 단점: 느리고 비싸며 대규모로 반복하기 어렵다.
- 원칙: 초기 골드셋과 모델 평가자의 기준을 맞추는 데 집중적으로 사용하고, 이후 표본 검수한다.

좋은 평가는 일반적으로 세 평가자를 적절히 섞는다.

```text
코드: 구조와 객관적 오류 검사
모델: 의미와 표현 품질 검사
사람: 정답과 평가 기준 자체를 검증
```

## 5. 결과가 매번 달라질 때 보는 지표

### `pass@k`

k번 실행했을 때 **한 번이라도 성공할 확률**이다.

아이디어 생성처럼 여러 번 시도해서 하나만 성공해도 되는 작업에 적합하다. 하지만 실제
서비스가 한 번만 실행한다면 `pass@1`이 더 중요한 지표다.

### `pass^k`

k번 실행했을 때 **모든 실행이 성공할 확률**이다.

사용자에게 매번 안정적인 결과를 보여줘야 하는 작업에 적합하다. 한 번은 잘하지만
반복할수록 결과가 흔들리는 에이전트는 이 점수가 낮아진다.

## 6. 처음 만드는 순서

1. 개발하면서 이미 반복 확인하던 사례 20~50개를 모은다.
2. 각 사례에 명확한 입력, 기대 결과, 실패 조건을 적는다.
3. “해야 하는 사례”와 “하면 안 되는 사례”를 균형 있게 넣는다.
4. 과제마다 독립된 환경에서 여러 번 실행한다.
5. 가능한 부분은 코드로 평가하고, 의미 평가는 모델과 사람이 보완한다.
6. 점수와 함께 실패 Transcript를 읽고 테스트와 평가자를 계속 수정한다.

두 명의 검토자가 같은 과제를 보고 서로 다른 정답을 낸다면 모델보다 먼저 과제 정의를
고쳐야 한다. 또한 올바른 절차를 따르면 반드시 통과할 수 있는 레퍼런스 정답을 하나
만들어두는 것이 좋다.

## 7. 흔한 실수

- 정답 사례만 넣고 “하면 안 되는 사례”를 넣지 않는다.
- 한 번 실행한 결과만 보고 모델이 좋아졌다고 결론 내린다.
- 결과보다 특정 도구 호출 순서와 내부 절차를 지나치게 강제한다.
- LLM 평가자의 점수를 사람과 비교해보지 않고 그대로 신뢰한다.
- 전체 평균 점수만 보고 어떤 유형에서 실패했는지 확인하지 않는다.
- 평가가 100%가 된 뒤 더 어려운 과제를 추가하지 않는다.
- 평가 스위트를 한 번 만들고 계속 유지보수하지 않는다.

## 한 문장 결론

AI 에이전트 평가는 모델에게 시험을 한 번 치르게 하는 일이 아니라, **명확한 과제와
다양한 평가자를 만들고 같은 시험을 반복하면서 제품의 품질 변화를 추적하는 개발
시스템**이다.
