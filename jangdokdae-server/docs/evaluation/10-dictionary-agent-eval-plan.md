# 장독대 경제용어 분리·설명 파이프라인 평가 계획

> 기반 개념: [AI 에이전트 평가 방법 — 짧고 쉬운 요약](./09-ai-agent-eval-summary.md)
>
> 적용 대상: 한국은행 경제금융용어 원문의 개별 용어 분리, 쉬운 설명 생성, 본문 매칭

## 요약

현재 장독대 용어 시스템은 복잡한 멀티턴 에이전트라기보다 **LLM이 포함된 단계형
파이프라인**에 가깝다. 그래도 출력이 비결정적이고, 한 사례를 고치면 다른 분류가
망가질 수 있으므로 에이전트 평가 방식이 잘 맞는다.

지금 해야 할 일은 789개를 먼저 생성하는 것이 아니다.

1. 대표 원문 20~30개를 사람이 분류해 작은 골드셋을 만든다.
2. 현재 분리 제안기를 같은 항목에 3번씩 실행한다.
3. 코드 평가, 모델 평가, 사람 검수를 조합한다.
4. 기준을 통과한 뒤 5~10개씩 실제 생성·승인한다.
5. 발견한 실패 사례를 회귀 테스트에 계속 추가한다.

주요 평가지표는 **분리 관계 정확도, 개별 용어 정확도, 별칭 정확도, 원문 근거 충실도,
반복 실행 일관성**이다.

## 1. 평가 대상

평가 스위트는 세 개로 분리한다.

### A. 용어 분리 평가

원문 제목과 정의를 받아 다음 중 올바른 관계를 선택하는지 평가한다.

```text
single
distinct_concepts
aliases
notation
```

또한 관계에 맞는 개별 용어와 별칭을 정확히 만들어야 한다.

### B. 쉬운 설명 평가

검수된 개별 용어와 한국은행 원문을 받아 다음 조건을 만족하는 설명을 만드는지 평가한다.

- 원문에 없는 내용을 추가하지 않는다.
- 개별 용어에 해당하는 내용만 설명한다.
- 주린이가 이해할 수 있는 1~3문장이다.
- 투자 권유, 전망, 최신 수치를 추가하지 않는다.
- 맞춤법과 문장이 자연스럽다.

### C. 본문 매칭 평가

승인된 용어가 실제 뉴스 본문에서 정확히 표시되는지 평가한다.

- 필요한 용어는 찾는다.
- 짧고 흔한 단어는 과도하게 표시하지 않는다.
- 별칭으로 등장해도 대표 용어와 연결한다.
- 한 이슈당 최대 5개, 같은 용어는 첫 등장 한 번만 표시한다.

본문 매칭은 대부분 코드 기반으로 결정되므로 회귀 테스트 비중을 높인다.

## 2. Task와 Outcome 정의

용어 분리 Task 한 건은 다음 형태로 저장한다.

```json
{
  "id": "seg-001",
  "source_term": "단리/복리",
  "raw_definition": "한국은행 원문 전체",
  "expected": {
    "relationship": "distinct_concepts",
    "units": [
      {"term": "단리", "aliases": ["Simple Interest"]},
      {"term": "복리", "aliases": ["Compound Interest"]}
    ]
  },
  "tags": ["distinct", "two-units", "easy"]
}
```

이때 모델의 마지막 답변만 평가하면 안 된다.

| 구분 | 장독대에서 기록할 내용 |
| --- | --- |
| Transcript | 모델명, 프롬프트 버전, 원문, 원출력, 정규화 결과, 검증 오류 |
| Outcome | 최종 relationship, term units, aliases, 검증 통과 여부 |
| Metrics | 지연시간, 호출 횟수, 토큰 사용량, 예상 비용 |

DB에 `proposed`가 저장됐다는 사실만으로 성공한 것은 아니다. 골드셋과 일치하고 원문
근거 검사를 통과한 최종 `Outcome`이 성공 기준이다.

## 3. 초기 골드셋 구성

처음에는 24개 정도면 충분하다. 정답은 AI가 아니라 사람이 확정한다.

| 유형 | 권장 수 | 후보 |
| --- | ---: | --- |
| `distinct_concepts` | 7 | 단리/복리, 명목금리/실질금리, 독점/과점, 평가절상/평가절하, 정규직/비정규직, 외채/대외채권, 최종수요/중간수요 |
| `aliases` | 5 | 환매조건부매매/RP/Repo, 바젤은행감독위원회/바젤위원회(BCBS), 경기조절정책/경제안정화정책 등 |
| `notation` | 4 | 산업연관표(I/O Tables), Treasury Bill(T/B), 원/위안 직거래시장 등 |
| `single` | 4 | 빅테크, 기준금리, 한국은행, 인플레이션 |
| 모호·고난도 | 4 | 노동생산성/노동생산성지수, 매입외환/환가료, 위험가중자산/위험가중치, 제조업생산능력/가동률지수 |

위 표의 분류는 골드 정답이 아니라 **라벨링 후보**다. 특히 `aliases`와 고난도 항목은
한국은행 원문을 읽고 사람이 확정해야 한다.

데이터는 반드시 균형을 맞춘다.

- 분리해야 하는 사례와 분리하면 안 되는 사례
- 2개 용어와 3개 이상 용어
- 한글명·영문명·약어
- 쉬운 사례와 모호한 사례
- 현재 뉴스에서 자주 등장하는 용어와 경계 테스트용 용어

## 4. 평가자 설계

### 코드 평가자

다음은 항상 코드로 검사한다.

- JSON 스키마가 맞는가
- relationship 허용값을 사용하는가
- `distinct_concepts`가 2개 이상인가
- 나머지 관계가 대표 용어 1개인가
- 용어와 별칭이 중복되지 않는가
- 용어 길이가 DB 제한을 넘지 않는가
- 제안한 용어·별칭이 원문으로 뒷받침되는가
- 원문에 없는 숫자·약어를 만들지 않았는가
- 골드 relationship과 일치하는가
- 골드 용어·별칭 집합과 얼마나 일치하는가

### 모델 평가자

코드로 판단하기 어려운 항목만 별도 루브릭으로 평가한다.

1. 개념을 잘못 합치거나 나누지 않았는가
2. 생략형 용어를 자연스럽고 정확하게 복원했는가
3. 쉬운 설명의 모든 주장이 원문에 근거하는가
4. 주린이에게 이해 가능한가
5. 맞춤법과 문장이 자연스러운가

한 모델이 모든 항목을 한꺼번에 평가하지 않도록 `분리 정확성`과 `설명 품질`을 별도
평가로 나눈다. 근거가 부족하면 억지로 점수를 주지 않고 `Unknown`을 허용한다.

### 사람 평가자

- 초기 24개 골드 라벨은 사람이 전부 확정한다.
- 첫 모델 평가 결과도 전부 읽어 평가자와 사람의 차이를 확인한다.
- 안정화 이후에는 새 배치의 10~20%를 표본 검수한다.
- 모델 평가와 사람 평가가 자주 다르면 자동 승인하지 않는다.

## 5. 점수와 통과 기준

### 용어 분리 점수

| 항목 | 배점 |
| --- | ---: |
| relationship 정확도 | 30 |
| 개별 용어 집합 정확도 | 35 |
| aliases 집합 정확도 | 20 |
| 원문 근거 충실도 | 15 |

다음은 점수와 무관한 즉시 실패 조건이다.

- 원문에 없는 용어 또는 약어 생성
- 분리하면 안 되는 `notation`을 여러 용어로 분리
- 서로 다른 개념을 하나의 aliases 그룹으로 병합
- JSON 또는 DB 제약 위반

초기 능력 평가 목표는 평균 85점 이상으로 잡되, 즉시 실패 사례가 남아 있으면 자동
승인을 시작하지 않는다.

### 쉬운 설명 점수

| 항목 | 배점 |
| --- | ---: |
| 원문 근거 충실도 | 40 |
| 개념 정확성 | 25 |
| 초보자 가독성 | 20 |
| 맞춤법·문체 | 15 |

원문에 없는 사실, 숫자, 전망, 투자 권유가 하나라도 있으면 즉시 실패한다.

## 6. 반복 실행과 지표

운영에서는 설명과 분리안을 한 번 생성해 사용하므로 **`pass@1`을 주지표**로 사용한다.
여러 번 돌려 하나만 건지는 `pass@k`가 높아도 실제 사용자 경험은 좋아지지 않는다.

중요 회귀 골드셋은 각 Task를 3번 실행해 **3번 모두 통과하는지**도 본다.

```text
주지표: pass@1
안정성 지표: pass^3
보조지표: 평균 점수, 유형별 정확도, 평균 지연시간, 평균 비용
```

전체 평균만 보지 않고 관계별 결과를 따로 낸다.

```text
distinct_concepts accuracy
aliases accuracy
notation accuracy
single accuracy
```

특히 `notation` 오분리는 over-splitting, `distinct_concepts` 미분리는
under-splitting으로 별도 집계한다.

## 7. 권장 하니스 구조

```text
evaluation/dictionary/
  tasks/
    segmentation_gold.jsonl
    definition_gold.jsonl
    matching_cases.jsonl
  graders/
    structure.py
    source_support.py
    set_metrics.py
    llm_rubric.py
  run_segmentation.py
  run_definition.py
  report.py

docs/evaluation/results/
  dictionary-eval-YYYY-MM-DD.md
```

각 Trial은 독립적으로 실행하고 DB를 직접 변경하지 않는다. 같은 모델명이라도 프롬프트
버전, 실행 시각, 모델 설정을 결과에 반드시 남긴다.

## 8. 개발·배포 게이트

### 개발 중

- 코드 기반 평가: 모든 변경에서 실행
- 소형 5개 LLM 스모크 평가: 프롬프트나 모델 변경 시 실행
- 전체 24개 × 3회 평가: 배치 승인 전 또는 모델 교체 전 실행

### 자동 승인 시작 조건

- 코드 기반 즉시 실패 0건
- 전체 평균 85점 이상
- `notation` 과분리 0건
- 중요 회귀 Task `pass^3` 기준 충족
- 모델 평가와 사람 평가가 충분히 일치

그전까지는 결과를 `proposed`로만 저장하고 사람이 `approved`로 바꾼다.

## 이번 단계의 권장 결정

다음 5~10개를 바로 생성·승인하기 전에, 최소한 `단리/복리`,
`환매조건부매매/RP/Repo`, `산업연관표(I/O Tables)`를 포함한 작은 골드셋부터 만든다.

동일한 기준의 여러 Task와 반복 Trial이 생겨야 프롬프트 변경 전후를 비교할 수 있다.
