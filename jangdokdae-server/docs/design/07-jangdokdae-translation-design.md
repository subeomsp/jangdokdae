## 1~4. 개요·의도·흐름·페이지 경험

`주린이 번역`은 뉴스 요약이 아니라 **입문자가 뉴스를 해석하는 힘을 기르는 학습형 콘텐츠**다. 파이프라인 위치: `뉴스 분석(Issue Docent) → [주린이 번역] → 사용자 화면`.

사용자 흐름:

```
이슈 선택 → 콘텐츠 읽기(용어 클릭·전문가의 시각) → 이해도 확인(퀴즈 페이지) → 다음 이슈 추천
```

페이지는 뉴스 상세가 아니라 **콘텐츠처럼 가볍게 소비하는 미니멀 구조** — 제목·본문 중심, 용어·전문가 시각은 읽기 흐름을 방해하지 않는 보조 요소.

## 5. 주요 기능 상세

### 5.1 주식 용어 풀이

용어 설명은 뉴스마다 생성하지 않고 서비스 차원 **주식 용어 사전**의 정의를 쓴다. 본문에 클릭 표시할 용어는 **LLM이 선별**:

| 기준 | 내용 |
|------|------|
| 사전 등록 | 등록된 용어만 |
| 도메인 의미 | 주식 의미로 쓰인 경우만 (일반 의미 제외) |
| 이해 필요성 | 주린이가 모를 가능성 높은 용어 우선 |
| 가독성 | **최대 5개**, 동일 용어는 첫 등장만 |

### 5.2 전문가의 시각

전문가가 뉴스에서 얻는 핵심 해석을 입문자 언어로 — 포인트 나열이 아니라 **본문의 어떤 표현·숫자를 근거로 그 해석이 나왔는지** 함께. 기본 1개, 복잡한 뉴스 최대 3개. 구조: `제목(해석 한 문장) + 해설(2~4문장) + 근거(highlight_ids)`.

### 5.3 본문 근거 강조

전문가의 시각 포인트 선택 시 연결된 본문 표현·숫자가 강조됨 — 문단 고정이 아니라 **span 단위**(`highlight_id`) 연결.

### 5.4 이해도 확인 퀴즈

본문 하단이 아니라 **별도 퀴즈 페이지**(버튼 이동). 3문항: ① 용어(해당 콘텐츠 노출 용어 중 1개) ② 뉴스 이해(전문가의 시각 포인트 기반) ③ 섹터 기초 상식(sector_tags 기반). 난이도 표시 없음, 완료 후 정답·해설 제공.

### 5.5 다음 이슈 추천

| 유형 | 수 | 기준 |
|---------|---|------|
| 관련 이슈 | 2 | 같은 종목·섹터·테마 |
| 관심 기반 | 1 | 온보딩 관심 종목·섹터 |
| 관련 없는 이슈 | 1 | 다양성 확보용 |

학습 진척 기반 추천은 초기 범위 제외.

## 6. 화면 구성

이슈 피드(실제 이슈 중심) → 콘텐츠 페이지(본문 + 용어 클릭 / 전문가 시각은 웹 사이드 패널·모바일 접이식 / 이해도 확인 버튼) → 퀴즈 페이지(분리) → 완료 페이지(결과·해설 + 다음 이슈 4개).

---

## 7. 콘텐츠 작성 기준

| 영역 | 기준 |
|------|------|
| 전문가의 시각 | 해석 한 문장 + 근거 2~4문장. "매수해야 한다"·"확실히 오른다" 금지 → "부담으로 작용할 수 있다"·"추가 확인이 필요하다" |
| 용어 설명 | 정확·입문자 언어·설명 안에 또 다른 주식 용어 최소화·특정 뉴스 맥락에 비종속·투자 판단 배제 |
| 퀴즈 | 암기·일반 상식 금지, 이해 확인 중심. 해설 필수. 오답은 그럴듯하지만 핵심을 빗나간 항목 |

(본문 번역 톤 규칙은 [06 §15](./06-news-analysis-design.md#15-주린이-번역-작성-규칙)가 정본.)

## 8. 데이터 명세

테이블 5종 (구현 시 ORM이 정본이 된다):

| 테이블 | 핵심 필드 | 역할 |
|--------|----------|------|
| `stock_terms` | `term`(unique) · `definition` · `category` | 서비스 용어 사전 |
| `term_highlights` | `docent_id` FK · `term_dict_id` FK · `span_id` · `position` | LLM 선별 용어 강조 |
| `expert_views` | `docent_id` FK · `order`(1~3) · `title` · `description` · `highlight_ids[]` | 전문가의 시각 |
| `quizzes` | `docent_id` FK · `quiz_type`(term/news/sector) · `question` · `options` JSONB · `explanation` · `order` | 퀴즈 3문항 |
| `issue_recommendations` | `source_docent_id` · `recommended_docent_id` · `rec_type`(related/interest/random) · `score` | 추천 결과 |

API: `GET /api/docent/{id}`(본문+용어+전문가 시각) · `/quiz` · `/recommendations`.

## 9. LLM 생성 파이프라인

```
Issue Docent 발행 → [TranslationChain]
  ├─ 용어 선별 (prompts/select_terms.yaml)   → term_highlights
  ├─ 전문가 시각 (prompts/expert_view.yaml)   → expert_views
  └─ 퀴즈 생성 (prompts/quiz_generate.yaml)   → quizzes
→ 추천 데이터 생성 → issue_recommendations
```

프롬프트는 YAML로 관리(CLAUDE.md 원칙) — 입력: 본문·L2 프레임·태그·용어 사전, 출력: 구조화 JSON.

## 10. 추천 알고리즘

| 유형 | 선정 기준 |
|------|----------|
| 관련 2개 | pgvector cosine 유사 Docent 중 sector/company(primary) 겹침, 유사도 내림차순, 기읽음 제외 |
| 관심 1개 | 관심 종목(company primary) 또는 관심 섹터 포함 최신 Docent |
| 무관 1개 | 현재와 섹터가 겹치지 않는 최신 Docent 중 랜덤 (다양성 보장) |

## 12. 개발 범위 및 추후 확장

초기: 이슈 피드·콘텐츠 페이지(용어+전문가 시각+근거 강조)·퀴즈 페이지(3문항)·추천 4개. 확장: 학습 기록·오답 복습 / 피드 개인화 / 시각 품질 피드백 / 용어 사전 검색 페이지 / 연속 읽기 / 학습 진척 추천.
