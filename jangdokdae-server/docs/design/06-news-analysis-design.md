## 1. 목적

```
수집 → 전처리 → 임베딩·클러스터링 → [뉴스 분석] → Issue Docent 생성
```

이슈로 묶인 클러스터(top_issues 인계 → [05 §6.2](./05-embedding-clustering-design.md#62-상위-이슈-선정영속화))를 받아 **분류 → 콘텐츠 생성 → 발행**. 산출물: 분류(L1·L2·L3) · 본문(4-head) · 연결 모듈 · Issue Docent · 퀴즈.

## 2. 기획 의도

주식 뉴스 하나는 여러 성격(정책+산업+테마+주가)을 동시에 갖고, 분류의 용도도 셋(사용자 카테고리 / LLM 요약 방향 / 추천·학습 태그)이라 단일 분류로는 전부 어정쩡해진다 → **3레이어 분리**.

## 3. 분류 구조 개요

| 레이어 | 개수 | 노출 | 역할 |
|--------|------|------|------|
| **L1** 뉴스 유형 | 5 | ✅ | 무엇에 대한 뉴스인가 |
| **L2** 해석 프레임 | 6 | ❌ 내부 | 어떤 관점으로 읽는가 — 본문 구조·번역·퀴즈의 기준 |
| **L3** 태그 | 복수 | 일부 | 추천·검색·학습 메타데이터 |

## 4. L1 뉴스 유형

5개 고정(6개 이상이면 인지 부하) — **기업** / **산업·테마**(산업+테마 통합, 구분은 L3) / **시장·수급**(자금 흐름 흡수) / **정책·규제**(거시경제 흡수) / **글로벌·이슈**.

## 5. L2 해석 프레임

6개 — **실적·재무** / **제품·기술·계약**(수주·계약 통합) / **산업·경쟁 구조** / **정책·규제 영향** / **리스크·이슈**(자본정책·공시 이벤트 흡수, 추후 분리) / **주가·수급**.

## 6. L1 × L2 매트릭스

30개 조합 중 **핵심 9개**만 본문 템플릿을 둔다: 기업×{실적·재무, 제품·기술·계약, 리스크·이슈} · 산업·테마×{산업·경쟁, 정책·규제 영향} · 시장·수급×주가·수급 · 정책·규제×정책·규제 영향 · 글로벌·이슈×{정책·규제 영향, 리스크·이슈}. 나머지 조합은 가장 가까운 핵심 조합의 head 세트를 재사용.

## 7. 충돌 조합 처리 룰

| 충돌 | 룰 |
|-----------|--------|
| 기업 vs 주가·수급 | 본문 다수가 주가 움직임이면 시장·수급, 펀더멘털이면 기업 |
| 정책 vs 주가·수급 | 정책 자체가 핵심이면 정책·규제, 시장 반응이 핵심이면 시장·수급 |
| 정책 vs 산업·경쟁 | 발표 1주 이내 = 정책·규제, 이후 영향 분석 = 산업·테마 |

LLM 분류 우선순위: 한 기업 사건 → 기업 / 정부·중앙은행 발표가 시작점 → 정책·규제 / 해외 사건 → 글로벌 / 지수·수급 → 시장·수급 / 산업 변화 → 산업·테마. **일반 원칙: 원인 사건 = L1, 결과·해석 = L2.**

## 8. L3 태그

| 태그 | 구조 | 역할 |
|------|------|------|
| `sector_tags` | 폐쇄형 (반도체·2차전지·바이오 등) | 섹터 필터·퀴즈 |
| `company_tags` | `{ticker, name, role: primary\|mentioned}` | 관심 종목·종목 페이지 |
| `theme_tags` | 개방형 (AI·비만치료제 등, 좁게 시작) | 트렌드 묶음 |
| `term_tags` | 폐쇄형 용어 사전 연동 | 클릭형 용어 풀이·퀴즈 |
| `difficulty` | 초/중/고급 (초기 내부 전용) | 초보 모드 필터 |

> **primary/mentioned 구분 필수** — "삼성전자 호실적에 SK하이닉스도 강세"에서 primary=삼성전자, mentioned=SK하이닉스. 이 구분이 빠지면 추천·연결 모듈 정확도가 흔들린다.

## 9. 분류 적용 기준

L1·L2 각 1개 필수(L2는 신뢰도 낮으면 `불확실` 허용), sector/company/term 태그 복수 필수, theme은 해당 시. LLM 자동 분류 + 저신뢰 케이스는 **검수 큐** 분리. 노출: L1·sector·company / 내부: L2·term·difficulty.

## 10. 콘텐츠 구조 — 본문과 연결 모듈

```
[이슈 본체 — head 4칸 고정]
  1. 무슨 일이에요              ← 공통 (L1이 말투만)
  2. 왜 중요한가요              ← L2가 결정
  3. ★오해 방지                 ← L2가 결정
  4. 앞으로 뭘 보면 될까요       ← L2 후속 체크포인트
─────────────────────────────
[연결 모듈 — 별도 블록, 여기서 처음 기업 등장]
  관련 섹터 → 관련 종목 → 기업 상세
```

**연결 모듈을 분리하는 이유**: 오해 방지·체크포인트를 다 읽은 뒤에야 종목이 나오므로 "이 뉴스 = 이 종목 사라"는 오해를 **구조적으로** 막는다 — 투자 추천이 아니라 읽는 관점을 학습시킨다는 정체성과 일치.

## 11. 콘텐츠 생성의 두 관점

**프롬프트 질문**(내부, LLM이 답할 의문문)과 **노출 head**(외부, 읽는 렌즈)는 같은 뼈대의 양면. 같은 L1×L2 → 같은 head 순서 → 사용자가 "이 유형은 이렇게 읽는다"를 반복 학습.

## 12. L2별 상세 — 프롬프트 질문·노출 head·후속 체크포인트

프롬프트 질문 설계 목적 5가지: 사실 확인·중요성·기대 vs 실제 구분·오해 방지·다음 체크포인트. ★ = 오해 방지 head.

| L2 | 프롬프트 질문 (3) | 노출 head (2·3번째 — 1·4는 공통) | 후속 체크포인트 |
|----|----------------|--------------------------|--------------|
| **실적·재무** | 핵심 숫자·변화 부문 / 컨센서스 대비 / 일회성 vs 구조적 | 시장 예상과 비교하면 · ★이게 계속될까요 | 다음 분기 가이던스, 동종업체 비교 |
| **제품·기술·계약** | 기존 대비 차이 / 매출 연결 경로 / 남은 관문(인증·양산·규제) | 기존과 뭐가 다른가요 · ★진짜 돈이 되려면 | 양산 일정, 첫 매출 인식, 후속 수주 |
| **산업·경쟁 구조** | 수요·공급·가격·점유율 변화 / 유리·불리한 기업 유형 / 사이클 vs 구조 | 왜 중요한가요 · ★잠깐인가요, 계속될 변화인가요 | 가격 추이, 재고, 경쟁사 캐파 |
| **정책·규제 영향** | 무엇이 바뀌나·확정 여부·시행 시점 / 이득·부담 측 / 선반영 vs 미확인 | 누구한테 이득·부담일까요 · ★확정인가요, 검토 중인가요 | 시행 시점, 시행령, 업계 대응 |
| **리스크·이슈** | 확정 vs 미확정 사실 / 영향 범위·크기 / 일회성 vs 구조적 | 얼마나 영향을 줄까요 · ★일시적인가요, 계속될 문제인가요 | 공식 입장, 후속 공시, 동종업체 영향 |
| **주가·수급** | 무엇이 얼마나 움직였나 / 기사가 제시한 원인 / 펀더멘털 확인 vs 사후 해석 | 왜 이렇게 움직였대요 · ★진짜 이유일까요, 끼워 맞춘 걸까요 | 다음 거래일 흐름, 펀더멘털 동반 여부 |

## 13. 9개 조합 → head 세트 매핑

head 세트는 L2가 결정하므로 9개 조합이 **6개 세트**로 모인다. L1은 열기 head 말투("무슨 정책이에요" / "해외에서 무슨 일이에요" — 글로벌은 head 2도 "우리 시장엔 어떻게 올까요")와 연결 모듈 모양만 바꾼다.

## 14. 연결 모듈

| L1 | 부채살 구조 |
|----|------------|
| 기업 | 주체 기업 + 같은 섹터 연관·경쟁 2~3 |
| 산업·테마 | 하위 섹터별 → 대표 기업 2~3 |
| 정책·규제 / 글로벌 | 수혜 섹터 vs 부담 섹터 분리 |
| 시장·수급 | 그날 주도 섹터·종목 |

데이터 소스: `sector_tags` + `company_tags(primary/mentioned)`. **카피 원칙: 추천이 아니라 길 안내** — "주목할 종목"이 아니라 "이 이슈와 관련된 종목".

## 15. 주린이 번역 작성 규칙

- **투자 권유 표현 금지**: ❌ 유망하다·사야 한다·기회다·수혜주·확실하다·급등 예상 → ✅ 가능성이 언급되고 있다·영향을 받을 수 있다·보도에 따르면
- **톤**: 단어는 쉽게, 사고는 어른답게(유아어 금지) · 확신 대신 구분("이런 면과 저런 면이 같이 있다") · 사실은 단정적으로, 해석은 조심스럽게
- **용어**: `term_tags` 용어는 본문에 다 녹이지 않고 클릭형 용어 풀이로 분리

## 16. 퀴즈 연계 방식

4종 — 독해(기사 연동) · 후속 체크포인트(기사 연동, L2 체크포인트 기반) · 용어(`term_tag` 사전) · 섹터 상식(`sector_tag` 사전). 오답 선택지는 그럴듯하지만 핵심을 빗나간 항목(예: "다음 반도체 뉴스에서 먼저 확인할 것" 정답=실제 수주·가이던스, 오답=주가 흐름·경쟁사 주가·CEO 인터뷰).

## 18. NewsAnalysisAgent 설계

EmbeddingClusterer 완료 후 트리거되는 **L2 — LangGraph 에이전트** ([00 2계층 모델](./00-workflow-airflow.md#2계층-오케스트레이션-모델)). 파이프라인에서 LLM이 흐름을 판단하는 유일한 단계. **MVP는 18.2 단일 에이전트, 18.3은 품질 미달 시 승급.**

### 18.1 도구

`fetch_clusters`(is_analyzed=False·top_issues) · `classify`(L1·L2·L3) · `generate_content`(4-head) · `generate_quiz` · `build_connection_module` · `publish_docent` · `enqueue_review`(저신뢰 검수 큐).

### 18.2 플로우 (LangGraph, MVP)

```
fetch → classify → generate_content → build_connection → generate_quiz
  → (저신뢰 있으면 enqueue_review →) publish → END
```

State: `clusters`·`classified_news`·`low_confidence_ids`·`generated_content`·`docent_ready`·`errors`.

### 18.3 멀티 에이전트 (슈퍼바이저-워커) — 승급 설계

단일 에이전트 품질(용어 누락·환각·맥락 부족)이 임계 미달일 때 IBM 계층적 manager-worker 패턴으로 승급:

```
Docent Supervisor (분해·위임·종합·게이트)
 ├─ 이슈요약 worker        (10줄 이내 — 멘토)
 ├─ 용어해설 worker        (장독대 차별점)
 ├─ 종목·기업 연결 worker  (RAG — 벡터DB에 없으면 "확실치 않음", 멘토 엣지케이스)
 ├─ 배경·맥락 worker
 └─ 품질검증 worker        (뉘앙스·환각 검증 → 통과/재생성)   ← 앞 4개 병렬, 검증이 종합
```

원칙: 워커 1개 = 작업 1개(멘토 "한 호출엔 하나의 task") · **약한 부분만** 워커로 분리해 "왜 멀티 에이전트인가"를 품질 데이터로 정당화 · 멀티는 토큰·지연·디버깅 비용이 커 MVP에선 쓰지 않음(멘토 "과도한 최적화 불필요").
